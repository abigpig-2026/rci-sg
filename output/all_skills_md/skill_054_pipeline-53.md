# Usage

要使用此流水线，您需要提供目标 GitHub 仓库的 `owner/repo` 格式的仓库名称作为核心输入。流水线将自动执行预定义的七个阶段，每个阶段调用 GitHub CLI (`gh`) 命令来获取特定的仓库信息。

**基本调用方式：**
```bash
# 启动流水线，监控指定的仓库
run-pipeline --repo <owner/repo>
```

**可用参数：**
*   `--repo` (必需): 指定目标仓库，格式为 `owner/repo`。
*   `--limit`: 限制某些列表查询的返回数量（例如，工作流运行列表）。
*   `--log-failed`: 启用失败日志记录。
*   `--jq`: 为支持 JSON 输出的命令提供自定义的 jq 过滤表达式。
*   `--json`: 强制以 JSON 格式输出。

# Examples

**示例 1：监控 `octocat/Hello-World` 仓库**
```bash
run-pipeline --repo octocat/Hello-World
```
此命令将启动流水线，持续监控 `octocat/Hello-World` 仓库的拉取请求检查、问题列表、特定 PR（示例中为 #55）的详情以及工作流运行状态。

**示例 2：监控仓库并限制工作流运行列表数量**
```bash
run-pipeline --repo microsoft/vscode --limit 5
```
此命令监控 `microsoft/vscode` 仓库，并在列出工作流运行时将结果限制为最近的 5 条。

**示例 3：获取 JSON 格式的原始数据**
```bash
run-pipeline --repo tensorflow/tensorflow --json
```
此命令以 JSON 格式输出每个阶段的原始数据，便于后续程序化处理。

# Pipeline Stages

该流水线按顺序执行以下七个阶段。每个阶段接收上游的上下文或参数，执行特定的数据提取任务，并将结果传递给下一阶段或作为最终输出的一部分。完成第七阶段后，流水线将回到第一阶段开始新一轮监控。

1.  **阶段 1：拉取请求检查状态查询**
    *   **命令**: `gh pr checks 55 --repo <owner/repo>`
    *   **功能**: 查询指定仓库中编号为 55 的拉取请求（PR）的所有状态检查（checks）结果。这用于评估该 PR 是否通过了必要的自动化测试或验证。
    *   **输出流转**: 结果（`result`）作为字符串输出，并流转到阶段 2。

2.  **阶段 2：问题列表获取 (第一次)**
    *   **命令**: `gh issue list --repo <owner/repo> --json number,title --jq '.[] | "\(.number): \(.title)"'`
    *   **功能**: 获取指定仓库的公开问题（issue）列表，并以简洁的 `编号: 标题` 格式输出。这提供了仓库当前待处理事项的概览。
    *   **输出流转**: 结果（`result`）作为字符串输出，并流转到阶段 3。

3.  **阶段 3：问题列表获取 (第二次)**
    *   **命令**: `gh issue list --repo <owner/repo> --json number,title --jq '.[] | "\(.number): \(.title)"'`
    *   **功能**: 与阶段 2 相同，再次获取问题列表。在持续监控的上下文中，这可用于检测两次迭代之间问题列表的变化（例如，新开或关闭的问题）。
    *   **输出流转**: 结果（`result`）作为字符串输出，并流转到阶段 4。

4.  **阶段 4：拉取请求详情查询 (第一次)**
    *   **命令**: `gh api repos/<owner/repo>/pulls/55 --jq '.title, .state, .user.login'`
    *   **功能**: 通过 GitHub API 直接查询编号为 55 的拉取请求的详细信息，提取其标题、状态（如 open, closed）和创建者。这提供了比基础 `gh pr view` 更灵活的数据筛选。
    *   **输出流转**: 结果（`result`）作为字符串输出，并流转到阶段 5。

5.  **阶段 5：拉取请求详情查询 (第二次)**
    *   **命令**: `gh api repos/<owner/repo>/pulls/55 --jq '.title, .state, .user.login'`
    *   **功能**: 重复阶段 4 的查询。在迭代监控中，可用于确认 PR 状态是否稳定或已发生变化。
    *   **输出流转**: 结果（`result`）作为字符串输出，并流转到阶段 6。

6.  **阶段 6：工作流运行列表**
    *   **命令**: `gh run list --repo <owner/repo> --limit 10`
    *   **功能**: 列出指定仓库最近的工作流（Actions）运行记录，默认限制为 10 条。这有助于了解仓库最近的 CI/CD 活动状态和成功率。
    *   **输出流转**: 结果（`result`）作为字符串输出，并流转到阶段 7。

7.  **阶段 7：拉取请求检查状态复核**
    *   **命令**: `gh pr checks 55 --repo <owner/repo>`
    *   **功能**: 再次查询 PR #55 的检查状态。与阶段 1 呼应，在一轮监控周期结束时复核其状态，可以用于判断在监控期间检查状态是否发生改变（例如，从 pending 变为 success）。
    *   **输出流转**: 结果（`result`）作为字符串输出。至此，一轮监控完成，流水线将自动返回到阶段 1，开始下一轮迭代。
