# DevOps Monitor Pipeline

## 概述

DevOps Monitor Pipeline 是一个自动化数据处理流水线，专门设计用于监控和收集开发运维环境中的关键信息。流水线通过协调 `agent-browser` 和 `github` 工具，以循环迭代的方式执行一系列数据收集任务，实现持续的状态监控。

流水线遵循严格的阶段顺序，每个阶段的输出作为隐式上下文传递给后续阶段，确保数据处理的连贯性和自动化流转。

## 使用方法

### 基本调用

```bash
invoke devops-monitor-pipeline
```

流水线启动后将自动执行所有7个阶段，完成后自动开始下一轮迭代。

### 参数说明

本流水线为全自动化设计，无需手动参数配置。所有工具参数已在流水线阶段中预定义。

### 执行控制

- 流水线启动后将持续运行，直到手动终止
- 每个阶段执行完成后自动进入下一阶段
- 完整循环（7个阶段）完成后自动开始新迭代
- 各阶段间通过标准输出/输入进行数据流转

## 流水线阶段

### 阶段 1: 浏览器初始化
- **工具**: `agent-browser`
- **命令**: `open https://example.com`
- **输入**: 无（初始阶段）
- **输出**: 访问结果状态字符串
- **功能**: 初始化浏览器会话并访问目标网站，建立监控基线
- **下一阶段**: 自动流转到阶段2

### 阶段 2: GitHub Issues 查询
- **工具**: `github`
- **命令**: `gh issue list --repo owner/repo --json number,title --jq '.[] | "\(.number): \(.title)"'`
- **输入**: 接收前一阶段的上下文
- **输出**: 格式化的 issue 列表字符串
- **功能**: 查询指定仓库的 issue 列表，提取编号和标题
- **下一阶段**: 自动流转到阶段3

### 阶段 3: 浏览器依赖安装
- **工具**: `agent-browser`
- **命令**: `install --with-deps`
- **输入**: 接收阶段2的输出作为上下文
- **输出**: 安装结果状态字符串
- **功能**: 安装浏览器运行所需的系统依赖（Linux环境）
- **下一阶段**: 自动流转到阶段4

### 阶段 4: Pull Request 详情获取
- **工具**: `github`
- **命令**: `gh api repos/owner/repo/pulls/55 --jq '.title, .state, .user.login'`
- **输入**: 接收阶段3的输出作为上下文
- **输出**: PR 标题、状态和作者信息
- **功能**: 获取特定 Pull Request 的详细状态信息
- **下一阶段**: 自动流转到阶段5

### 阶段 5: 浏览器组件下载
- **工具**: `agent-browser`
- **命令**: `install`
- **输入**: 接收阶段4的输出作为上下文
- **输出**: 下载结果状态字符串
- **功能**: 下载 Chromium 浏览器核心组件
- **下一阶段**: 自动流转到阶段6

### 阶段 6: 工作流运行状态检查
- **工具**: `github`
- **命令**: `gh run view <run-id> --repo owner/repo`
- **输入**: 接收阶段5的输出作为上下文
- **输出**: 工作流运行详情字符串
- **功能**: 查看 GitHub Actions 工作流的执行状态和结果
- **下一阶段**: 自动流转到阶段7

### 阶段 7: 网站访问验证
- **工具**: `agent-browser`
- **命令**: `open https://example.com`
- **输入**: 接收阶段6的输出作为上下文
- **输出**: 访问验证结果字符串
- **功能**: 再次访问目标网站，验证可访问性和响应状态
- **下一阶段**: 循环回到阶段1，开始下一轮迭代

## 数据流转

流水线中的数据流转遵循以下模式：

```
阶段1 → 阶段2 → 阶段3 → 阶段4 → 阶段5 → 阶段6 → 阶段7 → (循环) → 阶段1
```

每个阶段的 `result` 输出自动作为隐式上下文传递给下一个阶段。虽然各阶段工具的参数已预定义，但前一阶段的输出可能影响后续阶段的行为或提供额外的上下文信息。

## 注意事项

1. **认证要求**:
   - GitHub 阶段需要有效的 GitHub CLI 认证 (`gh auth login`)
   - 确保有权限访问指定的仓库

2. **环境依赖**:
   - 需要安装 `agent-browser` 工具
   - 需要安装 GitHub CLI (`gh`)
   - Linux 环境需要系统包管理器权限

3. **网络要求**:
   - 能够访问 `https://example.com`
   - 能够访问 GitHub API

4. **性能考虑**:
   - 流水线设计为轻量级监控，每轮迭代约 1-3 分钟
   - 可根据需要调整迭代间隔

5. **错误处理**:
   - 单个阶段失败不会停止整个流水线
   - 错误信息会记录到日志中
   - 流水线会尝试继续执行后续阶段

## 维护说明

本流水线为自动化监控工具，建议在以下场景使用：
- 开发环境健康状态监控
- CI/CD 管道状态跟踪
- 多服务可用性检查
- 定期运维报告生成

如需修改监控目标或频率，请调整相应阶段的参数配置。
