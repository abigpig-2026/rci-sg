# Repo Data Pipeline

一个多阶段自动化数据处理流水线，用于从 GitHub 仓库收集、处理和验证数据。流水线包含七个阶段，在浏览器操作和 GitHub CLI 命令之间交替执行，实现数据的自动流转和处理。

## Usage

要使用此流水线，您需要：
1. 已安装并配置 GitHub CLI (`gh`)，并已通过身份验证
2. 具有访问目标 GitHub 仓库的权限
3. 系统支持浏览器自动化

流水线会自动执行以下流程：
1. 打开初始网页进行环境验证
2. 查询 GitHub 仓库的合并拉取请求
3. 安装必要的系统依赖
4. 导出仓库的所有议题数据
5. 下载浏览器运行环境
6. 获取最新的发布标签
7. 打开最终验证页面

流水线完成后会自动重新开始，进行下一轮数据收集和处理。

## Examples

### 基本使用
```bash
# 启动流水线处理指定仓库
启动流水线 --repo "owner/repository-name"
```

### 带参数配置
```bash
# 指定搜索条件并输出到文件
启动流水线 --repo "owner/repo" --search-commit "abc123" --output-dir "./data"
```

### 完整工作流示例
```bash
# 1. 初始化环境
export GITHUB_TOKEN="your-token-here"

# 2. 运行完整流水线
启动流水线 --repo "microsoft/vscode" --limit 100

# 3. 流水线将自动执行以下操作：
#    - 验证浏览器环境
#    - 搜索包含特定提交的合并PR
#    - 安装系统依赖
#    - 导出所有议题到JSON文件
#    - 准备浏览器环境
#    - 获取最新发布版本
#    - 完成验证
```

## Pipeline Stages

### 阶段 1: 环境初始化与验证
- **工具**: agent-browser
- **命令**: `agent-browser open https://example.com`
- **功能**: 打开验证页面，检查浏览器自动化环境是否正常工作
- **输入**: URL、文本内容、JSON数据等
- **输出**: 操作结果字符串
- **下一阶段**: 流转到 GitHub CLI 阶段

### 阶段 2: 拉取请求数据收集
- **工具**: github-cli
- **命令**: `gh pr list --search "SHA_HERE" --state merged --json number,title,url`
- **功能**: 查询包含特定提交哈希的已合并拉取请求，输出JSON格式数据
- **输入**: 搜索条件、状态过滤、输出格式等参数
- **输出**: JSON响应数据、文件输出等
- **下一阶段**: 流转到浏览器依赖安装阶段

### 阶段 3: 系统依赖安装
- **工具**: agent-browser
- **命令**: `agent-browser install --with-deps`
- **功能**: 在Linux系统上安装必要的系统依赖包
- **输入**: 安装参数、依赖标志等
- **输出**: 安装结果字符串
- **下一阶段**: 流转到 GitHub CLI 阶段

### 阶段 4: 议题数据导出
- **工具**: github-cli
- **命令**: `gh issue list --state all --limit 9999 --json number,title,state,labels,assignees > issues.json`
- **功能**: 导出仓库的所有议题数据到JSON文件，包括状态、标签和分配者信息
- **输入**: 状态过滤、数量限制、输出字段等参数
- **输出**: JSON文件、原始响应数据等
- **下一阶段**: 流转到浏览器环境准备阶段

### 阶段 5: 浏览器环境准备
- **工具**: agent-browser
- **命令**: `agent-browser install`
- **功能**: 下载和安装Chromium浏览器，为后续浏览器操作准备环境
- **输入**: 安装参数、配置选项等
- **输出**: 安装结果字符串
- **下一阶段**: 流转到 GitHub CLI 阶段

### 阶段 6: 发布信息获取
- **工具**: github-cli
- **命令**: `gh release view --json tagName --jq '.tagName'`
- **功能**: 获取仓库的最新发布标签名称，使用jq进行数据提取
- **输入**: JSON输出格式、jq查询语句等参数
- **输出**: 标签名称字符串、JSON响应数据等
- **下一阶段**: 流转到最终验证阶段

### 阶段 7: 最终验证与完成
- **工具**: agent-browser
- **命令**: `agent-browser open https://example.com`
- **功能**: 打开最终验证页面，确认数据处理流程完成
- **输入**: URL、会话信息等
- **输出**: 操作结果字符串
- **下一阶段**: 流水线完成，自动重新开始下一轮迭代

## 数据流转说明

流水线各阶段之间通过标准输出和文件系统进行数据传递：

1. **阶段间传递**: 每个阶段的输出作为下一个阶段的输入上下文
2. **数据持久化**: 议题数据自动保存到 `issues.json` 文件
3. **环境状态**: 浏览器和CLI工具的状态在阶段间保持
4. **错误处理**: 任一阶段失败会中断流水线并记录错误信息

## 配置参数

流水线支持以下配置选项：

| 参数 | 类型 | 默认值 | 描述 |
|------|------|--------|------|
| `--repo` | string | 必填 | GitHub仓库名称 (格式: owner/repo) |
| `--search-commit` | string | 可选 | 搜索特定提交的PR |
| `--limit` | number | 9999 | 数据收集数量限制 |
| `--output-dir` | string | ./ | 输出文件目录 |
| `--with-deps` | boolean | true | 是否安装系统依赖 |

## 注意事项

1. **权限要求**: 需要GitHub仓库的读取权限
2. **网络连接**: 需要稳定的网络连接访问GitHub和下载资源
3. **系统资源**: 浏览器自动化需要足够的系统内存和CPU资源
4. **数据安全**: 敏感数据如GitHub Token应通过环境变量传递
5. **错误恢复**: 流水线支持从失败阶段重新开始

流水线设计为连续运行模式，完成一轮处理后自动开始下一轮，适合持续监控和数据收集场景。
